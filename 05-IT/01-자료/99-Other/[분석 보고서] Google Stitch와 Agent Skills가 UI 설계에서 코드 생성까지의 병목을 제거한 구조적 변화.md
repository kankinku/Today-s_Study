## 1. 문제 정의 또는 한계 상황

기존 UI 개발 프로세스는 **디자인 → 문서화 → 컴포넌트 구현**이 분리되어 있었다.  
디자이너 산출물(Figma 등)은 정적인 결과물에 머물고, 개발자는 이를 다시 해석해 코드로 옮겨야 했다. 이 과정에서 다음과 같은 구조적 문제가 반복적으로 발생했다.

- 디자인 의도와 실제 구현 간 불일치
    
- 디자인 시스템 문서(design spec)의 수동 작성 비용
    
- 컴포넌트 분해 및 재구성 과정에서의 시간 소모
    
- 초기 UI 스캐폴딩에 과도한 인력·시간 투입
    

결과적으로 단순 대시보드나 CRUD UI조차 **수 시간 단위의 반복 노동**이 필요했고, 이는 제품 초기 속도를 결정적으로 저하시켰다.

---

## 2. 도입된 기술 또는 구조

이 한계를 해소하기 위해 도입된 구조가 **Google Stitch + Agent Skills 조합**이다.

- **Stitch**
    
    - 프롬프트 기반 AI UI 디자인 도구
        
    - 단일 자연어 입력으로 모바일/웹 UI 생성
        
    - HTML + Tailwind CSS 코드 동시 출력
        
- **Agent Skills (Stitch Skills)**
    
    - Stitch 결과물을 후속 자동 처리하는 에이전트 스킬
        
    - 주요 스킬:
        
        - `design-md`: 디자인 시스템 문서 자동 생성
            
        - `react-components`: React 컴포넌트 변환
            
    - AST(Abstract Syntax Tree) 검증 포함
        

이 조합은 “디자인 산출물”을 중간 단계로 두지 않고, **설계 의도 → 실행 가능한 코드**로 바로 연결하는 구조를 형성한다.

---

## 3. 작동 메커니즘 (How)

전체 워크플로우는 다음과 같이 동작한다.

1. **자연어 입력**
    
    - 예: “마라톤 훈련 앱 대시보드”
        
2. **Stitch UI 생성**
    
    - 레이아웃, 컴포넌트 구조, 스타일 결정
        
    - 즉시 HTML/Tailwind 코드 출력
        
3. **Agent Skills 자동 후처리**
    
    - `design-md`: 색상, 컴포넌트, 레이아웃 규칙을 문서화
        
    - `react-components`: UI를 React 컴포넌트 단위로 분해
        
4. **AST 기반 검증**
    
    - 문법적 오류, 컴포넌트 구조 이상 여부 사전 차단
        
5. **즉시 개발 투입 가능 상태 도달**
    

이 과정은 CLI 한 줄 설치로 연결된다.

```
npx add-skill google-labs-code/stitch-skills --skill react:components --global
```

핵심은 사람이 개입하던 **해석·전환·정리 단계가 모두 자동화**되었다는 점이다.

## 4. 선택의 이유 (Why)

이 구조가 선택될 수밖에 없었던 이유는 명확하다.

- **기존 방식의 병목 지점이 ‘생각’이 아니라 ‘변환’이었기 때문**  
    UI 설계 자체보다, 설계를 코드로 옮기는 과정이 가장 비효율적이었다.
    
- **프롬프트 → 코드의 단일 경로 확보**  
    중간 산출물(Figma, Spec 문서)을 거치지 않아 정보 손실이 발생하지 않는다.
    
- **품질 보장의 자동화**  
    AST 검증을 통해 “빠르지만 불안정한 코드 생성”이라는 기존 AI 한계를 회피한다.
    
- **시간 단위가 아닌 분 단위 생산성**  
    수동 3시간 30분 작업이 3분 30초로 단축되는 60배 차이는 단순 효율 개선이 아니라 **개발 방식의 전환**을 의미한다.
    

대안(디자인 AI + 수동 컴포넌트화)은 여전히 사람의 재해석을 요구하며, 이 구조적 병목을 제거하지 못한다.

---

## 5. 구조적 시사점

Stitch + Agent Skills 조합은 단순한 도구가 아니라 **반복 가능한 개발 구조**를 제시한다.

- UI 초기 설계는 더 이상 “비용”이 아니라 “즉시 생성 가능한 자원”이 됨
    
- 디자인 시스템 문서는 결과물이 아니라 **자동 생성되는 부산물**로 전환
    
- 프론트엔드 개발자는 UI 구현자가 아니라 **의도 검증자·조정자 역할**로 이동
    
- 향후 멀티 에이전트 구조와 결합 시:
    
    - UI 생성 → 상태 관리 연결 → API 바인딩까지 연쇄 자동화 가능
        

결과적으로 이 구조는 **MVP 제작, 내부 도구, 실험적 제품 개발**에서 반복적으로 재사용될 수 있으며, UI 개발의 기준 시간을 근본적으로 재정의한다.
