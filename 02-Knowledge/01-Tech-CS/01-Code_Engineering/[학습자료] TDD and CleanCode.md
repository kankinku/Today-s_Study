# TDD and CleanCode

### 1. 핵심 개발 방법론 (이론적 배경)

AI와 협업할 때 코드의 품질을 높이고 컨텍스트(맥락)를 유지하기 위한 두 가지 핵심 개념을 설명합니다.

- **TDD (Test-Driven Development):**
    
    - **Red 페이즈:** 실제 코드가 없는 상태에서 실패하는 테스트 코드를 먼저 작성합니다.
        
    - **Green 페이즈:** 테스트를 통과할 수 있을 정도의 임시 코드를 구현합니다.
        
    - **Refactor 페이즈:** 코드의 가독성을 높이고 품질을 개선합니다.
        
    - **AI 협업 시 장점:** 작성된 테스트 코드가 강력한 컨텍스트가 되어, 세션이 길어져도 AI가 구현해야 할 기능을 명확히 파악할 수 있게 돕습니다.
        
- **Clean Architecture (클린 아키텍처):**
    
    - 서비스의 핵심 로직(도메인)을 중앙에 두고 외부 요소가 내부를 의존하게 설계하는 방식입니다.
        
    - **AI 협업 시 장점:** 의존성이 명확히 분리되어 있어, AI가 전체 코드를 다 읽지 않아도 특정 레이어 내에서 완성도 높은 코드를 작성할 수 있습니다.
        

### 2. 'Feature Planner' 스킬 상세

영상에서 무료로 배포하는 이 스킬은 약 300줄의 프롬프트로 구성되어 있으며 다음과 같은 기능을 수행합니다.

- **명시적 계획 수립:** "피처 플래너를 사용해 계획해 줘"라고 명령하면 작동하며, 사용자가 결정해야 할 기술 스택이나 구조를 먼저 물어봅니다.
    
- **품질 게이트 (Quality Gate):** 각 단계가 끝날 때마다 빌드, 린트, 테스트 통과 여부 등을 체크박스로 확인하여 품질을 강제합니다.
    
- **단계별 설계 (Phase-based Design):** 전체 기능을 7~11개 정도의 작은 단계로 나누어 설계하며, 각 단계마다 TDD 워크플로우를 적용합니다.
    
- **플랜 템플릿 사용:** `plan_template.md` 파일을 기반으로 마크다운 문서를 생성하여 특정 폴더에 저장하고 관리합니다.
    

### 3. 실제 작동 및 활용 방식

- **계획서 생성:** 프로젝트 개요, 아키텍처 결정(ADR), 단계별 구현 계획이 포함된 상세한 마크다운 파일이 자동으로 만들어집니다.
    
- **컨텍스트 유지:** 개발 도중 세션이 끊기거나 컨텍스트가 부족해지더라도, 생성된 플랜 파일만 참조하면 AI가 이전에 하던 작업을 그대로 이어갈 수 있습니다.
    
- **실행 명령:** 플랜 파일이 생성된 후에는 "첫 번째 페이즈를 구현해 줘"와 같이 간단한 명령만으로 TDD 원칙에 따른 코드 작성이 시작됩니다.
